Product Requirements Document: WFM Triage & Unit Economics Engine
1. Executive Summary & Business Case
Problem Statement
Uber operates at the intersection of digital algorithms and physical operations. Machine Learning (ML) models are deployed to handle high-volume tasks such as driver document verification and safety flag classification. These models cannot achieve 100% accuracy in unstructured real-world scenarios. Consequently, ambiguous edge cases are routed to human operators. This Human-in-the-Loop (HITL) fallback creates a linear cost structure. As operational volume scales, human review costs scale proportionally. Furthermore, presenting raw, unstructured data to human operators increases cognitive load, extending Average Handling Time (AHT) and degrading unit margins.

Solution
The WFM Triage & Unit Economics Engine is a production-ready software module designed to govern the intersection of AI confidence thresholds and human operational costs. It acts as an intelligent routing layer and an operational copilot. By dynamically managing when to utilize API inference versus human review, and by structuring the data presented to human operators, the engine minimizes operational expenditure while maintaining strict safety and compliance standards.

2. Value Proposition & Competitive Advantage
Unique Value
Standard ML deployments treat human operations as a static safety net. This engine treats human operations as a dynamic unit economic variable and a continuous data generation source. It mathematically calculates the trade-off between API inference costs and human labor rates to optimize routing logic.

Production Readiness
The tool is designed as a plug-and-play architecture. It utilizes a Multi-Agent System (MAS) pattern  that can interface with existing Uber data pipelines via standard API endpoints. It requires minimal backend refactoring, acting as a middleware layer between the core AI models and the frontend operator queues.

3. Core Features
3.1 Live HITL Router
Description: An automated triage gate for incoming safety and document flags.

Mechanics: The engine evaluates the confidence score generated by the upstream LLM. If the score exceeds a predefined threshold (e.g. 95%), the system executes Straight-Through Processing (STP) to auto-resolve the ticket. If the score falls below the threshold, the system routes the ticket to the human queue.


Value: Establishes rigid operational guardrails  preventing low-confidence AI decisions from impacting end users.

3.2 Dynamic Margin & Cost Simulator
Description: A forecasting tool for Workforce Management (WFM) to control unit economics.

Mechanics: Provides an interactive interface to adjust the AI Confidence Threshold for escalation. As the threshold changes, the system calculates the projected operational cost by balancing API inference expenses against human agent review time and hourly rates.

Value: Enables operations managers to make mathematical decisions regarding risk tolerance and budget burn rates.

3.3 Context-Engineered Operator View (Copilot)
Description: A structured interface designed to minimize operator cognitive load.

Mechanics: Upon escalation, the system uses Retrieval-Augmented Generation (RAG) to fetch the specific Standard Operating Procedure (SOP) relevant to the ticket. It presents the agent with a synthesized three-bullet summary of the anomaly, the governing policy, and a pre-drafted recommended action.


Value: Translates loosely defined requirements into clear jobs. Reduces AHT and prevents decision fatigue.

3.4 RLHF Capture Gate (Regulated Friction)
Description: A mechanism to capture high-signal data from human operators.

Mechanics: If an operator overrides the Copilot's recommended action, they must categorize the disagreement via a dropdown menu (e.g. Policy Change, Unseen Edge Case). This action packages the data packet and routes it to a designated Golden Dataset.


Value: Closes the feedback loop, transforming human operational work into structured training data for Reinforcement Learning from Human Feedback (RLHF).

3.5 Time-Lapse Simulator (Model Drift vs. Data Flywheel)
Description: A module to track and project model degradation and retraining schedules.

Mechanics: Tracks Model Accuracy, HITL Escalation Volume, and Daily WFM Cost over time. As real-world data shifts, it flags model drift. A deployment protocol then injects the RLHF Golden Dataset back into the model to restore accuracy and reduce human volume.

Value: Visualizes and manages the proprietary data flywheel, ensuring the system becomes more efficient over its lifecycle.

3.6 5-Phase Deployment Dashboard
Description: A programmatic tracker for global rollout.


Mechanics: Utilizes Earned Value Management (EVM)  principles to track project health across five distinct deployment phases.


Value: Provides central visibility into execution gaps and ensures delivery discipline.

4. Phased Implementation Approach
Deploying an AI-integrated operational tool requires a phased approach to balance system reliability with cost optimization.

Phase 1: Manual Baseline & Context Capture (Months 0 to 1)
Objective: Establish ground truth data and finalize SOPs.

Process: 100% of tasks are handled by human operators using standard interfaces. No AI routing is active.


Action: Operations Program Managers standardize decision trees and map edge cases to define exact job scope. Baseline metrics for AHT, QA score, and unit cost are established.

Phase 2: Shadow AI & Tooling Integration (Months 1 to 2)
Objective: Test the model without exposing customers to risk.

Process: The AI model processes tickets in parallel with human operators. The system logs the AI's intended decisions and confidence scores but does not execute them.

Action: Compare AI outputs to human resolutions. Calibrate the initial confidence thresholds required for Phase 3. Deploy the Context-Engineered Operator View in a controlled sandbox to measure AHT reduction.

Phase 3: Hybrid Routing Activation (Months 2 to 3)
Objective: Begin cost reduction while maintaining strict quality control.

Process: The Live HITL Router is activated. The confidence threshold is set to a highly conservative level (e.g. 98%). Tickets above 98% are auto-resolved. The rest are routed to humans utilizing the Copilot interface.

Action: Monitor SLA compliance and error rates. Activate the RLHF Capture Gate to begin building the Golden Dataset.

Phase 4: Dynamic Threshold Optimization (Months 3 to 4)
Objective: Maximize unit economics based on operational capacity.

Process: WFM utilizes the Dynamic Margin & Cost Simulator to adjust thresholds dynamically. If ticket backlog spikes or BPO capacity drops, the threshold is slightly lowered to accept more AI decisions, balancing cost against minor increases in error tolerance.

Action: Focus on reducing cognitive load for complex edge cases.

Phase 5: Automated Flywheel & Steady-State (Month 4+)
Objective: Continuous improvement and model retraining.

Process: The Time-Lapse Simulator metrics govern operations. As model drift occurs, the RLHF Golden Dataset generated by the agents is used to retrain the underlying model.

Action: The retrained model handles a broader scope of edge cases natively, pushing the required human headcount down and completing the data flywheel cycle.

5. Success Metrics (KPIs)
System Throughput (TCH): Tasks completed per hour, measuring total volume handled by STP and human agents.

Straight-Through Processing (STP) Rate: Percentage of total tickets resolved by the AI without human intervention.

Average Handling Time (AHT): Time spent by an operator per ticket, expected to decrease due to the Copilot interface.

Inter-Rater Reliability (IRR): Measurement of operator consensus to ensure SOP clarity and minimize arbitrary human decisions.

Cost Per Resolution (CPR): Blended calculation of API API inference costs and total WFM hourly expenditure.
